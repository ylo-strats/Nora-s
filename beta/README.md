# SecureDRM v2.0 — Secure Medical Document Viewer

Production-grade document licensing system with per-user device binding,
AES-256-GCM encrypted content, remote license validation, and admin control panel.

---

## Architecture

```
secure-drm/
├── backend/          Node.js/Express license server
│   ├── server.js     License API + admin API + content decryption gate
│   ├── package.json
│   └── data/
│       ├── db.json       User/device/ban database (auto-created)
│       ├── chunks.json   AES-encrypted content chunks (generated by ingest)
│       └── manifest.json Public section manifest (titles only)
├── viewer/
│   └── index.html    Standalone viewer template (USER_ID placeholder)
├── admin/
│   └── index.html    Admin dashboard (standalone, no build step)
├── tools/
│   ├── ingest.js          DOCX → AES chunks pipeline
│   ├── generate-packages.js  Per-user ZIP generator
│   └── package.json
└── dist/             Generated per-user ZIPs (git-ignored)
```

## Step 1 — Deploy the License Server

```bash
cd backend
npm install

# Set required environment variables:
export ADMIN_SECRET="your-strong-admin-secret-here"
export TOKEN_SECRET="your-strong-token-secret-here"
export CONTENT_KEY="exactly-32-chars-content-key!!!"   # must be 32 chars
export ALLOWED_ORIGINS="https://your-domain.com"
export DATA_DIR="/data"                                # Railway Volume mount for persistent users
export PORT=3001

npm start
```

**Deploy options:** Railway, Render, Fly.io, VPS, Docker — any Node.js host.

> ⚠️ The three secrets must be identical between the server and the ingestion tool.
> If CONTENT_KEY changes, re-run ingest to re-encrypt content.

**Railway persistence:** create a Railway Volume mounted at `/data`.
The server now automatically uses `/data/db.json` when that volume exists.
You can also set `DATA_DIR=/data` explicitly in Railway variables.
User IDs, registered devices, bans, font settings, and access counters are
stored in that `db.json`, so they survive restarts and redeploys.

If users disappear after redeploy, the Railway Volume is not mounted to this
service yet. Add a Volume in the Railway service settings, mount it as `/data`,
then redeploy once.

## Step 2 — Ingest your Document

```bash
cd tools
npm install

# From a DOCX file:
CONTENT_KEY="exactly-32-chars-content-key!!!" \
  node ingest.js --input ../my-study.docx --output ../backend/data/chunks.json

# From a pre-structured JSON file:
CONTENT_KEY="exactly-32-chars-content-key!!!" \
  node ingest.js --input ./sections.json --output ../backend/data/chunks.json
```

**sections.json format:**
```json
[
  { "id": "sec1", "title": "Section Title", "content": "Full text content..." },
  { "id": "sec2", "title": "Another Section", "content": "More content..." }
]
```

The ingestion pipeline outputs:
- `backend/data/chunks.json` — AES-256-GCM encrypted content (server-side only)
- `backend/data/manifest.json` — public manifest with titles only (bundled in viewer ZIPs)

## Step 3 — Seed Users on the Server

```bash
# Create USER-001 through USER-020 on the license server
curl -X POST https://your-server.com/api/admin/seed \
  -H "Authorization: Bearer your-admin-secret" \
  -H "Content-Type: application/json" \
  -d '{"count": 20}'
```

## Step 4 — Generate Per-User Packages

```bash
cd tools

# Generate ZIPs for specific users (server must be running):
LICENSE_SERVER="https://your-server.com" \
ADMIN_SECRET="your-admin-secret" \
  node generate-packages.js --users USER-001,USER-002,USER-003

# Generate ZIPs for a range AND register them on the server:
LICENSE_SERVER="https://your-server.com" \
ADMIN_SECRET="your-admin-secret" \
  node generate-packages.js --range 1 20 --register
```

Each user receives `dist/package_USER-001.zip` containing:
- `index.html` — viewer with `USER-001` hardcoded
- `manifest.json` — section index (no content)
- `README.txt` — instructions for the recipient

## Step 5 — Admin Dashboard

Open `admin/index.html` in any browser. Enter your server URL and admin secret.

**Capabilities:**
- View all issued users, active devices, access counts
- Ban / unban users (immediate effect on next ping)
- Revoke individual device slots (frees a slot for re-activation)
- Issue new user IDs
- Adjust max users and max devices per user globally
- Change the document font shown to users
- Monitor suspicious activity log (unknown packages, device limit violations, etc.)

## Security Model

| Threat | Mitigation |
|--------|-----------|
| Package forwarded to another person | Device fingerprint checked on every activation and 12h ping; 3rd device denied |
| Token stolen | Tokens expire after 13h; fingerprint re-validated on ping |
| Content extracted from package | All content served server-side only; no plaintext in the ZIP |
| Admin endpoint brute-forced | Rate limiting + Bearer token auth |
| DevTools console extraction | Soft deterrent (window size detection, right-click block) |
| Printing | CSS `@media print { display: none }` |

### What This Is NOT

This is DRM, not perfect copy-protection. A determined technical user with physical access
to a device can always extract content displayed on screen (screen recording, etc.).
The goal is to prevent casual forwarding and create legal accountability through
per-user ID watermarking.

## API Reference

### License Endpoints (called by viewer)

| Method | Endpoint | Body | Description |
|--------|----------|------|-------------|
| POST | `/api/license/activate` | `{userId, fingerprint}` | Register device, get signed token |
| POST | `/api/license/ping` | `{token, fingerprint}` | Refresh token (every 12h) |
| POST | `/api/license/chunk` | `{token, fingerprint, chunkId}` | Fetch decrypted content chunk |

### Admin Endpoints (require `Authorization: Bearer ADMIN_SECRET`)

| Method | Endpoint | Body | Description |
|--------|----------|------|-------------|
| GET | `/api/admin/stats` | — | Full stats dump |
| POST | `/api/admin/config` | `{maxUsers, maxDevicesPerUser}` | Update global limits |
| POST | `/api/admin/issue` | `{userId?}` | Issue new user package |
| POST | `/api/admin/ban` | `{userId, ban}` | Ban or unban user |
| POST | `/api/admin/revoke-device` | `{userId, fpHash}` | Remove device slot |
| POST | `/api/admin/seed` | `{count}` | Bulk pre-seed users |
| POST | `/api/admin/encrypt-chunk` | `{plaintext}` | Encrypt a single chunk (admin tool) |
| GET | `/health` | — | Health check |

## Upgrading from v1 (the scaffold)

The scaffold used:
- XOR + btoa "encryption" → **replaced with AES-256-GCM**
- In-memory client-side license DB → **replaced with server-side persistent DB**
- No token expiry → **HMAC-signed tokens expire in 13h**
- Weak fingerprint → **multi-signal fingerprint with persistent local salt**
- No 12h ping → **scheduled ping with offline grace**
- No content served server-side → **/api/license/chunk endpoint with full auth gate**
- Admin panel client-side only → **real admin API with Bearer auth + rate limiting**
